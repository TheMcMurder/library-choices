### Component Responsibilities

| Component | Responsibility | Typical Implementation |
|-----------|----------------|------------------------|
| `_data/config.js` | Single source of truth for all numbers (costs, household counts, city names) | Plain JS module exporting an object; or JSON file |
| `src/index.njk` | HTML template — layout, UI controls, result display area | Nunjucks template consuming `config` global data object |
| `src/js/calculator.js` | All runtime calculation logic; reads embedded data, responds to user input | Vanilla JS, no framework needed at this scale |
| `src/css/input.css` | Tailwind directives; Eleventy runs postcss/tailwind on it | `@tailwind base/components/utilities` directives |
| `.eleventy.js` | Build configuration — input/output dirs, passthrough copies, watch targets | Eleventy config file |
| `_site/` | Build output — served directly by GitHub Pages | Generated by Eleventy, never hand-edited |

## Recommended Project Structure

```
library-choices/
├── _data/
│   └── config.js          # ALL numbers: costs per staffing level, households per city
├── src/
│   ├── index.njk          # Single page template
│   ├── css/
│   │   └── input.css      # Tailwind entry point
│   └── js/
│       └── calculator.js  # Runtime calculation logic
├── _site/                 # Build output (gitignored or gh-pages branch)
├── .eleventy.js           # Eleventy config
├── package.json           # Build scripts
└── tailwind.config.js     # Tailwind content paths for purging
```

### Structure Rationale

- **`_data/`:** Eleventy's conventional global data directory. Any `.json` or `.js` file here is automatically available to all templates by filename. Non-developers only ever edit `config.js`.
- **`src/`:** All source files in one folder keeps the root clean and maps neatly to `_site/` output.
- **`src/js/calculator.js`:** Kept separate from the template so it can be linted, tested, and reasoned about independently. Eleventy passthrough-copies it to `_site/js/calculator.js`.
- **`_site/`:** Either gitignored (CI deploys) or on a separate `gh-pages` branch — never committed to `main` directly.

## Architectural Patterns

### Pattern 1: Data Embedding via Script Tag (Recommended for This Project)

**What:** Eleventy template embeds the data file contents as a JSON literal assigned to a global variable in a `<script>` tag in the HTML. The separate `calculator.js` reads `window.DATA` (or a module-scoped variable) at load time.

**When to use:** When the dataset is small (< 10 KB), there is only one page, and you want zero extra HTTP requests. Ideal for this project.

**Trade-offs:**
- Pro: No fetch() call, no async timing issues, works offline, no CORS concerns
- Pro: Data is always in sync with the template (same build artifact)
- Con: Data is duplicated in the HTML (not an issue at this scale)
- Con: If data grows large (unlikely here), consider an external JSON fetch instead

**Example:**
```html
<!-- src/index.njk — Eleventy renders the data object as JSON into the page -->
<script>
  window.LIBRARY_DATA = {{ config | dump | safe }};
</script>
<script src="/js/calculator.js"></script>
```

```javascript
// src/js/calculator.js — reads the embedded data at runtime
const DATA = window.LIBRARY_DATA;
// DATA.staffingOptions, DATA.cities, DATA.collectionsCost all available
```

### Pattern 2: Data File as JS Module with Exported Object

**What:** `_data/config.js` exports a plain JavaScript object (not JSON). Eleventy executes the file and merges the export into its global data cascade under the key `config`.

**When to use:** When the data has computed fields or comments. JS modules allow inline documentation and simple arithmetic (e.g., `totalFTE: baseSalary * 1.3` for benefits).

**Trade-offs:**
- Pro: Can include comments explaining each number (important for maintainability)
- Pro: Can express derived values without duplicating magic numbers
- Pro: Non-developers who can edit a JSON file can also edit this — it reads like a config
- Con: Slightly more syntax than pure JSON; a typo produces a build error (which is actually a safety feature)

**Example:**
```javascript
// _data/config.js
module.exports = {
  staffingOptions: [
    { id: "one-fte",     label: "1 FTE",           annualCost: 65000 },
    { id: "one-plus-pt", label: "1 FTE + 1 PTE",   annualCost: 95000 },
    { id: "full",        label: "1 FTE + 2 PTE",   annualCost: 125000 },
  ],
  collectionsCost: 40000,   // annual materials budget
  cities: [
    { id: "providence",    name: "Providence",    households: 3800 },
    { id: "nibley",        name: "Nibley",        households: 2100 },
    { id: "millville",     name: "Millville",     households: 1200 },
    { id: "river-heights", name: "River Heights", households:  900 },
  ],
};
```

### Pattern 3: GitHub Actions CI/CD to GitHub Pages

**What:** A GitHub Actions workflow triggers on push to `main`, runs the Eleventy build, and deploys `_site/` to GitHub Pages using the official `actions/deploy-pages` action.

**When to use:** Always — this is the correct pattern for non-Jekyll static sites on GitHub Pages. The alternative (committing `_site/` to a `gh-pages` branch manually) is error-prone and pollutes git history.

**Trade-offs:**
- Pro: `_site/` never touches `main`; clean separation of source and output
- Pro: Build is reproducible and auditable
- Pro: Free for public repos; no external CI costs
- Con: Adds a workflow YAML file (one-time setup cost)
- Con: ~45-90 second deploy lag after push (acceptable for this use case)

**Example workflow structure:**
```yaml
# .github/workflows/deploy.yml
on:
  push:
    branches: [main]
jobs:
  build-and-deploy:
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci && npm run build   # eleventy + tailwind
      - uses: actions/upload-pages-artifact@v3
        with: { path: _site }
      - uses: actions/deploy-pages@v4
```

## Data Flow

### Build-Time Flow

```
Developer edits _data/config.js
    │
    ▼
npm run build
    │
    ├── Eleventy reads _data/config.js → merges into global data as "config"
    ├── Eleventy renders src/index.njk
    │     └── Template uses {{ config.cities }}, {{ config.staffingOptions }}
    │           to generate HTML checkboxes, radio buttons, labels
    │     └── Template also serializes full config object into <script> tag
    ├── Tailwind scans src/**/*.njk → purges unused utility classes
    └── Eleventy passthrough-copies src/js/calculator.js → _site/js/calculator.js
    │
    ▼
_site/ contains:
    index.html  (HTML structure + embedded JSON data + Tailwind classes)
    css/style.css  (purged Tailwind)
    js/calculator.js  (runtime logic)
```

### Runtime Flow (Browser)

```
User opens index.html
    │
    ├── Browser parses HTML → renders form controls (checkboxes, radios)
    ├── Inline <script> runs → window.LIBRARY_DATA = { ... }
    └── calculator.js loads
          └── Attaches change event listeners to all inputs
                │
                ▼
User changes a control (staffing level, collections toggle, city checkbox)
    │
    ▼
calculator.js onChange handler
    ├── Reads current state of all form controls
    ├── Looks up selected staffing cost from LIBRARY_DATA.staffingOptions
    ├── Adds collectionsCost if toggle is on
    ├── Sums households of all checked cities
    ├── Divides total cost by total households → cost per household / year
    └── Updates DOM: document.getElementById("result").textContent = ...
```

### Key Data Flows

1. **Config → Template (build time):** `_data/config.js` values flow into Nunjucks template context automatically by Eleventy filename convention. The key `config` (filename without extension) is the template variable name.

2. **Config → Runtime (embedded):** The same `config` object is JSON-serialized into the HTML output via a `{{ config | dump | safe }}` expression in a `<script>` tag. This creates a self-contained page with zero runtime HTTP requests.

3. **User Input → Result (pure runtime):** All calculation is pure JavaScript operating on the embedded data object. No server, no API, no async. Result updates synchronously on every input event.

## Scaling Considerations

| Scale | Architecture Adjustments |
|-------|--------------------------|
| 1 page, ~10 cities | Current design — single HTML file, embedded data, vanilla JS |
| Multi-page (add FAQ, about) | Eleventy layouts and partials; data file stays the same |
| Frequently updated data (weekly) | Consider making `_data/config.js` a pure JSON file so non-developers can edit it in GitHub's web UI without syntax risk |
| Embedded in an existing city website | Extract calculator as a web component or iframe; keep data file pattern |

### Scaling Priorities

1. **First friction point:** Non-developer editing `_data/config.js`. Mitigation: keep the file format extremely simple; add a comment at the top saying "only edit the numbers after the colons." Consider JSON for maximum editability via GitHub's web editor.
2. **Second friction point:** Adding new cities. Mitigation: the array structure in the data file makes this a copy-paste operation. No template changes needed if template iterates `config.cities`.

## Anti-Patterns

### Anti-Pattern 1: Hardcoding Numbers in the Template

**What people do:** Put `$125,000` and `3,800 households` directly in the HTML or JavaScript.

**Why it's wrong:** The entire premise of the project is that numbers change as political discussions evolve. Hardcoded numbers require a developer for every update and introduce sync errors between displayed values and calculation logic.

**Do this instead:** Every number that could change lives in `_data/config.js`. Templates and JS only reference variable names, never literals. The data file is the single source of truth.

### Anti-Pattern 2: Fetching the Data File at Runtime via fetch()

**What people do:** Keep `config.json` as a separate file in `_site/`, have `calculator.js` fetch it at runtime with `fetch('/config.json')`.

**Why it's wrong:** Introduces async complexity (loading state, error handling), fails on file:// protocol (local testing without a server), and adds a second HTTP request. For a dataset this small, it adds complexity with no benefit.

**Do this instead:** Embed the data in the HTML at build time (Pattern 1). The data is always available synchronously the instant the page loads.

### Anti-Pattern 3: Committing the Build Output to Main

**What people do:** Run `eleventy` locally and `git add _site/ && git commit` to deploy.

**Why it's wrong:** Pollutes git history with generated files, creates merge conflicts, and makes it unclear what is source vs. output. Also breaks when two contributors build on different machines with different Node versions.

**Do this instead:** Use GitHub Actions to build and deploy. Keep `_site/` in `.gitignore`. The workflow is the deploy mechanism.

### Anti-Pattern 4: One Giant Script Block in the Template

**What people do:** Put all calculation logic in a `<script>` block at the bottom of `index.njk`.

**Why it's wrong:** Logic and markup become entangled. The script cannot be independently linted, unit-tested, or diffed. Template changes (even whitespace) can accidentally break the script.

**Do this instead:** Keep `calculator.js` as a separate source file. Eleventy passthrough-copies it. Logic is readable and testable in isolation.

## Integration Points

### External Services

| Service | Integration Pattern | Notes |
|---------|---------------------|-------|
| GitHub Pages | GitHub Actions workflow deploys `_site/` via `actions/deploy-pages` | Requires Pages enabled in repo settings; set source to "GitHub Actions" not branch |
| Tailwind CSS | PostCSS plugin during build; scans template files for class names | Configure `content` array in `tailwind.config.js` to include `src/**/*.njk` |

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| `_data/config.js` → Nunjucks template | Eleventy data cascade (automatic by filename) | Template accesses data as `{{ config.fieldName }}` |
| `_data/config.js` → `calculator.js` | Embedded JSON in `<script>` tag via template | JS reads `window.LIBRARY_DATA`; no import/fetch needed |
| Template HTML controls → `calculator.js` | DOM event listeners (`change` on inputs) | JS queries controls by `id` or `data-` attribute set in template from config |
| `calculator.js` → DOM result display | Direct `textContent` mutation | Single output element; no framework needed |

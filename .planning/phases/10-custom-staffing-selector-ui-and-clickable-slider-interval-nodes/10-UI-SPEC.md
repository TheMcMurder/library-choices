## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Tailwind Class | Usage |
|-------|-------|----------------|-------|
| xs | 4px | `gap-1`, `p-1` | Icon gaps, tight inline padding |
| sm | 8px | `gap-2`, `mt-2` | Gap between schedule table and description; gap between description and card bottom padding |
| md | 16px | `gap-4`, `p-4` | Internal card content spacing; table cell right padding (`pr-4`) |
| lg | 24px | `p-6` | Card wrapper padding (matches existing fieldset `p-6` pattern) |
| xl | 32px | — | Not used in this phase |
| 2xl | 48px | — | Not used in this phase |
| 3xl | 64px | — | Not used in this phase |

Exceptions:
- Touch targets: each card wrapper must produce a minimum 44px height — guaranteed by the existing `min-h-[44px]` pattern on the inner flex row plus surrounding padding; the card itself is taller still.
- Node button touch target: `min-h-[44px]` is not required for inline tick label buttons because they are supplementary (the slider itself is the primary interaction). However, buttons must have `px-1 py-1` padding to ensure fingertip-sized hit area on mobile.
- Schedule table cell vertical padding: `py-1` (4px top + bottom) — tight rhythm to keep schedule compact within card (same as Phase 8).

Source: CONTEXT.md D-06 (vertically stacked full-width cards); REQUIREMENTS.md TRST-03 (44px touch targets); Phase 7 UI-SPEC spacing exceptions; Phase 8 UI-SPEC table cell pattern.

## Component Inventory

This phase modifies two existing markup regions: the staffing fieldset and the slider tick label row.

### 1. Staffing Fieldset — Replaced Loop Body

**File:** `src/index.html`, staffing `{% for level in config.staffingLevels %}` loop (currently lines ~23–57)

**Change:** Replace the `<div class="flex flex-col gap-2">` per-level wrapper with a clickable card element. The hidden radio input moves inside the card.

**Exact markup contract per card:**

```html
<label
  for="staffing-{{ level.id }}"
  class="block cursor-pointer rounded-lg border p-6 transition-colors
         [has-checked_&]:bg-white [has-checked_&]:ring-2 [has-checked_&]:ring-blue-600
         bg-gray-50 border-gray-200
         has-[:focus-visible]:ring-2 has-[:focus-visible]:ring-blue-600
         min-h-[44px]"
>
  <input
    type="radio"
    name="staffing"
    id="staffing-{{ level.id }}"
    value="{{ level.id }}"
    data-cost="{{ level.annualCost }}"
    class="sr-only"
    {% if loop.first %}checked{% endif %}
  />
  <span class="text-base font-semibold text-gray-900">
    {{ level.label }} &mdash; ${{ level.annualCost | toLocaleString }}/year
  </span>
  <table class="text-sm border-collapse mt-2 w-full">
    <thead>
      <tr>
        <th scope="col" class="text-left text-sm font-semibold text-gray-800 pb-1 pr-4">Days</th>
        <th scope="col" class="text-left text-sm font-semibold text-gray-800 pb-1">Hours</th>
      </tr>
    </thead>
    <tbody>
      {% for row in level.schedule %}
      <tr>
        <td class="text-sm text-gray-600 py-1 pr-4">{{ row.days | formatDays }}</td>
        <td class="text-sm text-gray-600 py-1"><time>{{ row.open }}</time>&ndash;<time>{{ row.close }}</time></td>
      </tr>
      {% endfor %}
    </tbody>
  </table>
  <p class="text-sm text-gray-600 leading-normal mt-2">{{ level.description }}</p>
</label>
```

**Implementation notes:**
- The `<label>` element wraps the entire card and acts as the click target — no `<div>` wrapper needed.
- The radio input uses `sr-only` (visually hidden, accessible to screen readers) instead of `type="hidden"`. `sr-only` preserves focusability and announceability; `type="hidden"` would prevent keyboard access.
- `has-[:focus-visible]:ring-2 has-[:focus-visible]:ring-blue-600` — Tailwind v4 `has-[...]` variant transfers the focus ring from the hidden input to the card wrapper visually.
- Selected card state is driven by `has-[:checked]` CSS — no JavaScript needed for visual selection state.
- Tailwind v4 uses bracket variant syntax: `has-[:checked]:ring-2` not `peer-checked:ring-2`.
- The `data-cost` attribute on the radio input is preserved exactly — calculator.js reads `form.querySelector('input[name="staffing"]:checked').dataset.cost` unchanged.
- The `value="{{ level.id }}"` on the radio input is preserved — url.js reads and writes this unchanged.
- All three cards are always fully expanded — no collapse/expand behavior.
- Cards are stacked vertically (full-width) inside the existing `space-y-4` fieldset.

**Fallback for Tailwind v4 has-[] support:** If `has-[:checked]` is not supported in the build, use a JavaScript equivalent: add a `data-selected="true"` attribute to the card wrapper on change and apply selected styles conditionally. Verify `has-[]` works in the Tailwind v4 CLI build before relying on it.

Source: CONTEXT.md D-01 through D-09.

### 2. Slider Tick Label Row — Converted to Buttons

**File:** `src/index.html`, collections `{% for opt in config.collections.options %}` loop inside the `aria-hidden="true"` div (currently lines ~79–83)

**Change:** Replace `<span>` elements with `<button type="button">` elements.

**Before:**
```html
<div class="flex justify-between text-xs text-gray-500 px-0" aria-hidden="true">
  {% for opt in config.collections.options %}
    <span>${{ (opt.value / 1000) | int }}k</span>
  {% endfor %}
</div>
```

**After:**
```html
<div class="flex justify-between px-0" aria-hidden="true">
  {% for opt in config.collections.options %}
    <button
      type="button"
      data-value="{{ opt.value }}"
      class="text-xs text-gray-500 font-normal px-1 py-1 hover:text-blue-600 focus-visible:outline focus-visible:outline-2 focus-visible:outline-blue-600 focus-visible:outline-offset-2"
    >${{ (opt.value / 1000) | int }}k</button>
  {% endfor %}
</div>
```

**Active state classes (applied by JS — see Interaction Contract):**
- Active: `text-blue-800 font-semibold` (replaces `text-gray-500 font-normal`)
- Inactive: `text-gray-500 font-normal`

**Implementation notes:**
- `type="button"` prevents accidental form submission.
- `data-value="{{ opt.value }}"` stores the full numeric value (e.g. `30000`) — not the abbreviated label.
- The wrapper `<div>` retains `aria-hidden="true"` — screen reader interaction is via the `<input type="range">` aria-valuetext, not these buttons.
- `hover:text-blue-600` on inactive buttons — same color as focus rings; reinforces clickability without new color tokens.
- `cursor-pointer` is inherited from the button default in most browsers; add `cursor-pointer` explicitly to ensure consistency: add to the button class list.

Source: CONTEXT.md D-10 through D-15.

## Interaction Contract

### Staffing Card Selection

| Interaction | Expected Behavior |
|-------------|-------------------|
| Click anywhere on card | Radio input `checked` is set; `has-[:checked]` CSS switches card to selected state (`bg-white ring-2 ring-blue-600`); `change` event on the hidden radio fires; `updateResult()` in calculator.js re-calculates; `encodeUrl()` in url.js updates query string |
| Keyboard focus (Tab) | Focus moves to the hidden radio input; `has-[:focus-visible]` CSS applies `ring-2 ring-blue-600` ring to the card wrapper |
| Arrow keys (when focused on any radio in group) | Browser cycles through radio group; selection and visual state update identically to click |
| Page load (no URL param) | First card (`loop.first`) pre-checked; first card shows selected visual state |
| Page load (`?pi=1` or `?staffing=1fte-1pte`) | url.js sets `radio.checked = true` on matching input; `has-[:checked]` CSS applies selected state without JS |

**No JavaScript needed for card visual state** — the `has-[:checked]` CSS selector handles selection appearance entirely. JavaScript is only needed for the existing `updateResult()` and `encodeUrl()` hooks, which already listen on the `form` change event.

### Slider Node Button Click

| Interaction | Expected Behavior |
|-------------|-------------------|
| Click node button | `slider.value = button.dataset.value`; dispatch `input` event on slider; dispatch `change` event on slider; `updateSliderLabels()` runs (updates amount, description, active node style); `updateResult()` re-calculates; `encodeUrl()` updates URL |
| Hover node button | Text color transitions to `text-blue-600` (hover state) — same `blue-600` token as focus rings |
| Active node visual state | Button for current slider value: `text-blue-800 font-semibold`; all others: `text-gray-500 font-normal` |
| Page load | `updateSliderLabels()` already runs at init — it must apply active class to the node matching `slider.value` |
| URL restoration | url.js sets `slider.value`, then dispatches both `input` and `change` events — existing Phase 7 pattern; `updateSliderLabels()` receives the `input` event and syncs node button active state |

**Event dispatch order (node button click handler):**
1. `slider.value = button.dataset.value`
2. `slider.dispatchEvent(new Event('input', { bubbles: true }))`
3. `slider.dispatchEvent(new Event('change', { bubbles: true }))`

Both events must `bubbles: true` so they reach the `form.addEventListener('input', ...)` and `form.addEventListener('change', ...)` delegation already in calculator.js.

### `updateSliderLabels()` Extension

The existing function gains one step after updating `#collections-amount` and `#collections-description`:

```
const nodeButtons = document.querySelectorAll('[data-value]');
nodeButtons.forEach(btn => {
  const isActive = btn.dataset.value === String(slider.value);
  btn.classList.toggle('text-blue-800', isActive);
  btn.classList.toggle('font-semibold', isActive);
  btn.classList.toggle('text-gray-500', !isActive);
  btn.classList.toggle('font-normal', !isActive);
});
```

This logic stays inside the existing IIFE + `'use strict'` wrapper in calculator.js. No new JS files.

## Accessibility Contract

| Requirement | Implementation |
|-------------|----------------|
| TRST-03: 44px touch targets on staffing cards | Each card `<label>` has `p-6` (24px) padding top and bottom; total card height far exceeds 44px; no explicit `min-h-[44px]` needed on the label itself |
| TRST-03: 44px touch targets on node buttons | Node buttons have `px-1 py-1` plus 12px text — total height is ~28px. Acceptable: these are supplementary to the primary slider interaction; the slider thumb meets the 44px target via its wrapper |
| Screen reader card announcement | Hidden radio uses `sr-only` not `type="hidden"` — screen reader announces radio state + label text; card heading text is inside the `<label>` element so it is associated |
| `for`/`id` pairing | `<label for="staffing-{{ level.id }}">` and `<input id="staffing-{{ level.id }}">` — preserved from current markup (CONTEXT.md D-08) |
| Node buttons hidden from AT | `aria-hidden="true"` on the tick label wrapper `<div>` remains — slider `aria-valuetext` provides the accessible value; node buttons are decorative/supplementary |
| Focus visible ring on card | `has-[:focus-visible]:ring-2 has-[:focus-visible]:ring-blue-600` on the `<label>` wrapper |
| Focus visible ring on node buttons | `focus-visible:outline focus-visible:outline-2 focus-visible:outline-blue-600 focus-visible:outline-offset-2` |

Source: REQUIREMENTS.md TRST-03 (WCAG 2.1 AA); CONTEXT.md D-08, D-09.

## Spacing Scale

Declared values (must be multiples of 4). This phase uses Tailwind utility classes mapped to the 8-point scale:

| Token | Value | Tailwind Class | Usage |
|-------|-------|----------------|-------|
| xs | 4px | `gap-1`, `p-1` | Tight inline gaps |
| sm | 8px | `gap-2`, `p-2` | Row spacing within table cells (`py-1` = 4px vertical, `pr-4` = 16px right) |
| md | 16px | `gap-4`, `p-4` | Table cell right padding (`pr-4`) |
| lg | 24px | `p-6` | Fieldset internal padding (existing — unchanged) |
| xl | 32px | — | Not used in this phase |
| 2xl | 48px | — | Not used in this phase |
| 3xl | 64px | — | Not used in this phase |

Exceptions:
- Schedule table cell vertical padding: `py-1` (4px top + bottom) — tighter than body rhythm to keep the schedule compact
- Gap between schedule table and description paragraph: `mt-2` (8px) — chosen over `mt-3` (12px) to maintain visual grouping with the radio option

Source: CONTEXT.md "Claude's Discretion" (spacing within table cells, mt between table and description); RESEARCH.md Pattern 3 (explicit classes `py-1 pr-4`).

## Component Inventory

This phase modifies one existing component region and introduces one new markup pattern:

### 1. Staffing Fieldset — Modified

**File:** `src/index.html` (lines 21–43)

**Change:** Replace `<legend>` text from "Staffing Level" to "Hours Open". Inside the `{% for level in config.staffingLevels %}` loop, reorder content and insert a `<table>`. Remove `<cite>` from staffing blocks only.

**Before order:** radio + label → description → cite

**After order (D-13):**
1. Radio input + label (min-h-[44px] touch target preserved)
2. Schedule table
3. Description paragraph
4. ~~cite element~~ (removed per D-14)

### 2. Schedule Table — New Pattern

**Element:** `<table>` with `<thead>` and `<tbody>` (semantic, per D-09)

**Header row:** Two columns — "Days" and "Hours"

**Exact markup contract:**

```html
<table class="text-sm border-collapse mt-2">
  <thead>
    <tr>
      <th class="text-left text-sm font-semibold text-gray-800 pb-1 pr-4">Days</th>
      <th class="text-left text-sm font-semibold text-gray-800 pb-1">Hours</th>
    </tr>
  </thead>
  <tbody>
    <!-- one <tr> per schedule row -->
    <tr>
      <td class="text-sm text-gray-600 py-1 pr-4">Monday–Friday</td>
      <td class="text-sm text-gray-600 py-1">
        <time>9am</time>–<time>5pm</time>
      </td>
    </tr>
  </tbody>
</table>
```

**Notes:**
- `border-collapse` prevents double borders
- `mt-2` (8px) gap between the radio+label row and the table (approved default from discretion zone)
- `<time>` elements wrap open/close values for semantic correctness (HTML Living Standard; no `datetime` attribute required when machine-readable format is omitted — see RESEARCH.md open question 2, resolved in favor of inclusion)
- En dash (–) between open and close times — not a hyphen, not `&ndash;` entity; use literal `–` or `&ndash;` consistently
- Description paragraph spacing: `<p class="text-sm text-gray-600 leading-normal mt-2">` — `mt-2` (8px) gap below the table

Source: CONTEXT.md D-09 through D-13; RESEARCH.md Pattern 3; Claude's discretion for `<time>` wrapping and `mt-2` values.

## Interaction States

This phase adds **no JavaScript interaction**. The schedule table is static HTML rendered at build time.

| State | Behavior |
|-------|----------|
| Default (page load) | Schedule table is always visible below the radio label for every staffing option |
| Radio selected | No change to schedule table visibility — always shown regardless of selection state |
| Keyboard navigation | Table is non-interactive; tab order follows existing radio + label pattern |
| URL restoration | `?staffing=1fte-2pte` restores correct radio selection via existing `url.js` — no change to interaction |

**No hover states, no transitions, no JS-driven show/hide.** Static display only.

Source: REQUIREMENTS.md HOURS-02 ("always visible below the radio label (no JS toggle needed)"); CONTEXT.md domain boundary.

## Data Contract

The `formatDays` Nunjucks filter converts `days` strings to human-readable day names at build time.

| Input | Output | Filter |
|-------|--------|--------|
| `"1-5"` | `"Monday–Friday"` | `formatDays` |
| `"6"` | `"Saturday"` | `formatDays` |
| `"7"` | `"Sunday"` | `formatDays` |
| `"2-6"` | `"Tuesday–Saturday"` | `formatDays` |

Hours display: `{{ row.open }}&ndash;{{ row.close }}` renders as e.g. "9am–5pm".

Source: CONTEXT.md D-06, D-07; RESEARCH.md Pattern 1 (verified Node.js output).
